# PRD — 学童保育Webアプリ（がくどう）

## 1. Overview

学童保育施設向けのWebアプリケーション。施設のHP機能、保護者への写真・連絡事項の共有、QRコードによる生徒の入退場記録を一つのプラットフォームに統合する。現在LINEとKAZASUに分散している業務を集約し、先生の業務負担軽減と保護者の利便性向上を実現する。対象は1施設・生徒30名・先生6名・保護者約30家庭の規模。

## 2. Goals

- KAZASUを解約し、入退場管理の固定費を削減する
- 先生がLINEで個別に行っている情報共有をアプリに集約し、業務負担を軽減する
- 保護者がスマホで子どもの登園状況・連絡事項・活動写真を手軽に確認できるようにする
- HPを刷新し、説明会案内など新規保護者向けの情報発信を機能させる

## 3. Non-goals

- **動画共有**: ストレージ・配信コストが無料枠と相性が悪い。v1.1以降で検討
- **延長保育料金の自動計算・請求**: 入退場時間の記録さえあれば手動で計算可能
- **複数施設対応**: 現在1施設のみ。需要が顕在化してから対応
- **保護者同士のコミュニケーション**: LINEが十分に機能している
- **プッシュ通知**: ブラウザでの確認で十分。技術的複雑性が高い
- **連絡事項の既読管理**: 表示するだけで見落としは十分改善される
- **OAuth / ソーシャルログイン**: 先生が初期アカウントを作成する運用で十分

## 4. Target users / personas

### 一般訪問者（未ログイン）
学童保育への入会を検討している保護者。説明会の日程や施設の基本情報を知りたい。スマホ・PC両方でアクセスする。現状は更新されていないHPしかなく、説明会がいつあるかわからないという声がある。

### 保護者（ログインユーザー）
学童に通う子どもの親。約30家庭、各家庭1〜2名。スマホで手軽に子どもの登園確認、連絡事項の確認、活動写真の閲覧をしたい。現状はすべてLINEで情報を受け取っており、過去の情報が流れてしまい探しにくい。

### 先生（管理者ユーザー）
学童保育のスタッフ6名。生徒の入退場管理、連絡事項の配信、写真のアップロードを行う。スマホ・PC・タブレットを利用。現状はKAZASU（入退場）とLINE（その他）を併用しており、ツールが分散して手間がかかっている。

## 6. Functional requirements

### FR-1: 公開HP
- 施設紹介、説明会案内、お知らせの各セクションが表示される
- 先生がお知らせを更新すると一般訪問者に最新情報が表示される
- スマホ（375px）で横スクロールなく全コンテンツが読める

### FR-2: 認証・ログイン
- メールアドレス＋パスワードによるログイン
- ログイン後、ロール（先生 / 保護者）に応じたダッシュボードにリダイレクト
- 誤ったパスワードではエラーメッセージを表示しログイン不可
- セッションはブラウザを閉じても一定期間保持（7日間）

### FR-3: 保護者アカウント管理
- 先生がメールアドレス・初期パスワード・紐付ける生徒を指定してアカウントを作成
- 1つの保護者アカウントに複数の生徒を紐付け可能
- パスワードリセット: メールアドレス入力 → リセットメール送信 → リンク経由で新パスワード設定（有効期限24時間）

### FR-4: 生徒マスター管理
- 先生が生徒の名前・クラス情報を登録・編集できる
- 保護者ロールからは生徒管理ページにアクセス不可

### FR-5: QRコード生成・カード印刷
- 生徒ごとにユニークなQRコードを生成
- 名刺サイズのカード用PDFをダウンロード（生徒名 + QRコード）
- 複数生徒を選択して一括PDF出力

### FR-6: QRコード入退場記録
- WindowsノートPC搭載のWebカメラでQRコードを読み取り（ブラウザ経由）
- 1回目のスキャンで「入場」、2回目で「退場」を記録
- 未登録のQRコードはエラー表示、記録されない
- オフライン時はオフラインである旨を画面に表示

### FR-7: 入退場履歴閲覧
- 先生: 全生徒の入退場記録を日付で絞り込んで一覧表示（生徒名・入場時刻・退場時刻）
- 保護者: 自分の子どもの当日の入退場記録のみ表示、他の生徒の記録は見えない

### FR-8: 連絡事項
- 先生: タイトルと本文を入力して連絡事項を登録・編集。新しい順に一覧表示
- 保護者: 連絡事項を新しい順に一覧閲覧、タップで全文表示

### FR-9: 写真共有
- 先生: 写真を複数枚同時にアップロード。アップロード日時の新しい順にサムネイル一覧表示
- 保護者: サムネイル一覧 → タップでフルサイズ表示。グリッドレイアウト
- 未ログインユーザーが写真URLに直接アクセスした場合はログイン画面にリダイレクト

### FR-10: ロールベースアクセス制御
- 先生ロール: 全機能にアクセス可能
- 保護者ロール: 連絡事項閲覧、写真閲覧、自分の子どもの入退場履歴のみ
- 未ログイン: 公開HPのみ

## 7. Non-functional requirements

| 項目 | 要件 |
|------|------|
| レスポンス時間 | ページ表示 2秒以内（3G回線、写真サムネイル除く） |
| 同時接続 | 50ユーザー（30家庭 + 6先生 + バッファ） |
| 可用性 | 99%（月間ダウンタイム7時間以内。平日日中の利用が中心） |
| 写真ストレージ | 1枚あたり最大5MB。アップロード時にリサイズ（長辺1920px以下） |
| セキュリティ | パスワードはハッシュ化保存。認証済みセッションのみ保護ページにアクセス可能。HTTPS必須 |
| アクセシビリティ | スマホ（375px以上）でのレスポンシブ表示。入退場画面はWindowsノートPC（ブラウザ）での全画面表示を想定 |
| データ保持 | 入退場記録は最低1年間保持。写真は無期限（ストレージ枠内） |

## 8. Technical constraints

- 開発者はReact + TypeScript、SpringBoot + Kotlinの実務経験あり
- 予算はできる限り無料枠で運用（ホスティング・DB・ストレージ含む）
- 子どもの写真を扱うため、認証なしでのアクセスは絶対に不可
- フレームワーク・ホスティング先の選定はtech-selectionで決定する

## 9. Data model sketch

```
User
  - id
  - email
  - passwordHash
  - role: "teacher" | "parent"
  - name
  - createdAt

Student
  - id
  - name
  - className
  - qrCode (unique token)
  - createdAt

ParentStudent (many-to-many)
  - parentId -> User
  - studentId -> Student

AttendanceRecord
  - id
  - studentId -> Student
  - checkInAt: timestamp
  - checkOutAt: timestamp | null
  - date

Announcement
  - id
  - title
  - body
  - authorId -> User (teacher)
  - createdAt
  - updatedAt

Photo
  - id
  - url (storage path)
  - thumbnailUrl
  - uploadedBy -> User (teacher)
  - createdAt

PublicNotice (HP用お知らせ)
  - id
  - title
  - body
  - publishedAt
  - authorId -> User (teacher)
```

## 12. Risks & mitigations

| # | Risk | Impact | Mitigation |
|---|------|--------|------------|
| 1 | WindowsノートPCのWebカメラでQRコード読み取り精度が低い | 入退場記録が取れず、KAZASUを解約できない | M2完了後にQR読み取りのPoC実施。精度が低い場合は外付けカメラやUSBバーコードリーダーを検討 |
| 2 | 無料枠のストレージ容量が写真で埋まる | 写真共有機能が停止する | リサイズで1枚あたり容量を抑制。月次で使用量を監視し、枠を超える前に有料プランへの移行判断 |
| 3 | 保護者がアプリに移行せずLINEを使い続ける | 二重管理になり先生の負担がむしろ増える | M3（入退場）で先に運用移行し、アプリの利用を習慣化してからM4・M5で情報共有を移行 |
| 4 | パスワード管理に不慣れな保護者がログインできない | サポート負荷が先生に集中する | 初期パスワードは先生が直接伝達。パスワードリセットを初期から実装。FAQ的なログインヘルプを公開HPに掲載 |
| 5 | 子どもの写真が外部に漏洩する | 信頼の失墜、法的リスク | 認証必須の徹底。写真URLは推測不可能なランダムパスを使用。直接URLアクセスでもログイン画面にリダイレクト |

## 14. Open questions (resolved)

1. **入退場の端末**: ~~タブレットか？~~ → WindowsノートPC（KAZASU用の既存端末を流用）。PCに搭載されたWebカメラでQRコードを読み取る。タブレットではなくPC＋ブラウザでの動作を前提とする。
2. **メール送信**: ~~全員がメールアドレスを持っているか？~~ → 全保護者がメールアドレスを所持。キャリアメールのフィルタリングには注意が必要。
3. **HP上の施設紹介コンテンツ**: ~~誰が用意するか？~~ → 先生が素材を準備可能。
4. **QRカードの物理仕様**: ~~ラミネートか？~~ → プリンターで印刷しラミネート加工、または印刷所に依頼。PDF出力はA4用紙に複数枚面付けする形式が望ましい。
5. **先生アカウントの初期作成**: ~~手順は？~~ → 未定。初回デプロイ時にCLIまたはシードスクリプトで初期管理者アカウントを1つ作成し、その管理者が他の先生アカウントを画面上で作成する運用とする。
