# PRD — gakudo（学童保育Webアプリ）

## Status: Approved (2026-04-20)

---

## 1. Overview

gakudoは、父母運営型の学童保育施設向けの統合Webアプリケーションである。保護者・先生・管理者の3つのロールで利用し、QRコードによる児童の入退場管理、連絡事項・写真・資料の共有、延長保育料金の自動計算・請求確認、および外部向けHPを一つのプラットフォームで提供する。現在バラバラに運用しているKAZASU（入退場管理）、LINE（連絡）、紙（請求）、放置HP（外部情報発信）を統合し、コスト削減と業務効率化を実現する。

## 2. Goals

- KAZASUを解約し、月約3,000円の入退場管理コストを削減する
- 先生の事務作業（連絡配信・請求計算・請求配布）を自動化し、児童と過ごす時間を増やす
- 保護者が隙間時間にスマホ1つで子供の状況（入退場・連絡・写真・請求）を確認できるようにする
- 入園希望者が施設情報・説明会日程を簡単に見つけられるようにし、法人化後の新規顧客獲得を支援する
- 延長保育料金の手計算・紙運用を廃止し、計算ミスと配布の手間をゼロにする

## 3. Non-goals

- **オンライン決済：** クレカ・口座振替による保育料支払い機能は作らない。請求の「確認」のみ
- **チャット機能：** 保護者同士、保護者⇔先生の双方向メッセージは作らない。配信は一方向
- **出席簿・学習記録：** 児童の学習進捗や成績の管理は対象外
- **マルチテナント：** 複数施設の統合管理は対象外。1施設のみ
- **ネイティブアプリ：** iOS/Androidアプリは作らない。PWA/レスポンシブWebで対応
- **多言語対応：** 日本語のみ
- **動画共有：** ストレージコストの制約上、v1は写真のみ
- **グループ管理：** 役員会・係別のグループ分け機能はv2以降。LINEで継続
- **年度引継ぎ：** 役員入替の自動化はv2以降

## 4. Target users / personas

### 保護者（メインユーザー） — 「忙しいお母さん」
共働きで忙しい母親。スマホを主デバイスとして通勤中や昼休みの隙間時間に利用する。子供が学童に無事着いたかリアルタイムで知りたい。連絡事項やイベント写真を手軽にチェックしたい。ITに詳しくはないが、LINEやスマホアプリは日常的に使う。

### 先生（指導員） — 「限られた時間で事務をこなす先生」
児童の帰宅後にPCまたはスマホから連絡事項をまとめて配信する。スマホで撮った写真を気軽に共有したい。施設PCでQRコードを読み取り入退場を管理。月末に延長保育料金の計算・請求作業がある。事務作業を減らして児童と向き合う時間を確保したい。

### 管理者 — 「先生兼任の施設管理者」
児童・保護者・先生のアカウント管理、QRコード発行、延長保育料金のルール設定、外部HPの更新を担当。先生が兼任する可能性が高い。来年度の法人化に向けて外部への情報発信を強化したい。

### 入園希望者 — 「情報を探している親」
子供の学童入園を検討中。施設の情報や説明会日程をWebで調べたい。ログイン不要で公開HPを閲覧する。

## 6. Functional requirements

### FR-1: アカウント管理
- 管理者が保護者・先生のアカウントを作成できる（メール＋名前＋ロール）
- 招待メールが送信され、受信者がパスワードを設定してアクティベートする
- 重複メールアドレスでの登録を拒否する
- AC: 管理者が新規ユーザーを登録 → 招待メール送信 → 受信者がパスワード設定 → ログイン可能

### FR-2: 認証
- メール＋パスワードによるログイン
- セッション保持（7日間）
- パスワードリセット（メールリンク、24時間有効）
- AC: 正しい認証情報 → ログイン成功。不正 → エラー表示。リセットリンク24h超過 → 期限切れ表示

### FR-3: 児童・保護者紐付け
- 管理者が児童（名前・学年）を登録できる
- 児童と保護者アカウントを紐付けできる（1児童に複数保護者、1保護者に複数児童）
- AC: 紐付け後、保護者のダッシュボードに児童が表示される

### FR-4: QRコード発行
- 児童ごとにユニークなQRコードを生成
- 個別・一括でPDF出力（A4、児童名付き）
- 再発行時は旧コードを無効化
- AC: 生成 → PDF出力 → 印刷可能。再発行 → 旧コード読取でエラー

### FR-5: QRコード入退場記録
- PCのウェブカメラでQRコードを読み取り、入室/退室を自動判定
- 当日未入室の児童 → 入室記録。入室済み → 退室記録
- 無効コードはエラー音付きで拒否
- 読み取りから表示まで3秒以内
- AC: QRかざす → 3秒以内に児童名＋入室/退室表示。無効QR → エラー音＋メッセージ

### FR-6: 手動入退場フォールバック
- 児童一覧から選択して入室/退室を手動記録
- 手動記録には「手動入力」ラベル付与
- AC: 手動入力 → 「手動入力」ラベル付きで保存。履歴で区別表示

### FR-7: 入退場通知
- 児童の入室/退室時に紐付き保護者に通知
- 記録から60秒以内に通知到達
- 通知方法：プッシュ通知（対応環境）またはメール（フォールバック）
- 保護者が通知方法を選択可能（プッシュ/メール/両方/OFF）
- AC: 入室記録 → 60秒以内に保護者の選択した方法で通知到達

### FR-8: 入退場履歴閲覧
- 保護者が自分の子供の入退場履歴を閲覧
- 直近30日をデフォルト表示、月別切替可能
- 日付ごとに入室時刻・退室時刻・滞在時間を表示
- AC: 履歴画面 → 日付順一覧表示。月切替 → フィルタ適用

### FR-9: 連絡事項配信
- 先生がタイトル＋本文で連絡事項を投稿 → 全保護者に配信
- 保護者の閲覧を既読として記録
- 先生側で既読/未読人数を確認可能
- AC: 投稿 → 全保護者に配信。閲覧 → 既読カウント増加

### FR-10: 連絡通知
- 連絡事項投稿時に保護者に通知（5分以内）
- 通知タップで該当連絡に直接遷移
- プッシュ不可時はメールフォールバック
- AC: 投稿 → 5分以内に通知。タップ → 該当連絡表示

### FR-11: 写真共有
- 先生がスマホから写真をアップロード（自動圧縮：長辺1920px以下、500KB以下）
- 保護者がギャラリーでサムネイル閲覧＋拡大表示
- 日付・イベント名でフィルタ可能
- ストレージ80%到達時に管理者へ警告メール
- AC: アップロード → 圧縮保存。ギャラリー → サムネイル一覧。フィルタ → 絞込み表示

### FR-12: 写真公開/非公開管理
- デフォルトは「保護者限定（非公開）」
- 管理者が個別に「HP公開可」に変更可能
- HP公開中の写真を非公開に戻すと即座にHPから非表示
- AC: 公開設定変更 → HPに即時反映/非表示

### FR-13: 資料掲示板
- 先生がPDF等（最大10MB）をカテゴリ付きでアップロード
- カテゴリ別フィルタ＋タイトル検索
- 保護者はスマホでブラウザ内PDF表示
- AC: アップロード → カテゴリ付きで保存。フィルタ → カテゴリ絞込み。PDF → ブラウザ内表示

### FR-14: 延長保育料金ルール設定
- 管理者が通常保育終了時刻、延長料金単価、計算単位（30分等）を設定
- 設定変更は変更日以降に適用（過去の計算に影響しない）
- AC: ルール設定 → 保存成功。以降の計算に反映。過去データ不変

### FR-15: 延長保育料金自動計算
- 月末に児童ごとの延長保育利用回数・合計時間・金額を自動計算
- 先生が確認後「確定」→ 保護者に請求公開
- 手動入力の入退場記録もQR記録と同等に計算に含む
- AC: 月末 → 自動計算一覧表示。確定 → 保護者に公開。手動記録 → 計算に含まれる

### FR-16: 月次請求オンライン確認
- 保護者がアプリ上で延長保育の利用明細と合計金額を確認
- 過去月の請求履歴を月別に閲覧可能
- 請求確定時に保護者へ通知
- AC: 請求画面 → 当月明細表示。月切替 → 過去履歴表示。確定 → 通知到達

### FR-17: CSVエクスポート
- 管理者が期間指定で入退場データをCSVダウンロード
- 含む項目：児童名、日付、入室時刻、退室時刻、記録方法
- UTF-8 BOM付き（Excel文字化け防止）
- AC: 期間指定 → CSV出力 → Excelで文字化けなし

### FR-18: 外部向けHP
- 管理者が施設名・住所・電話番号・説明文を編集・公開
- お知らせ（説明会日程等）を追加可能（新着順表示）
- HP公開可の写真をギャラリーに表示
- ログイン不要で閲覧可能
- レスポンシブ対応（スマホでも見やすい）
- AC: 編集 → 即時公開反映。未ログイン → 閲覧可能。375px幅 → 横スクロールなし

## 7. Non-functional requirements

- **レスポンス速度：** ページ読み込みはLCP 2.5秒以内（3G回線想定）
- **QR読み取り速度：** かざしてから入退場表示まで3秒以内
- **通知遅延：** 入退場通知は記録から60秒以内、連絡通知は投稿から5分以内
- **可用性：** 月間稼働率99.5%以上（入退場管理は児童安全に直結）
- **同時接続：** 保護者30人＋先生3人＋管理者1人の同時アクセスに耐える
- **セキュリティ：** パスワードはbcryptハッシュ化。セッショントークンはhttpOnly cookie。HTTPS必須
- **プライバシー：** 児童の写真は認証済みユーザーのみアクセス可能。公開/非公開の切替は即時反映
- **アクセシビリティ：** WCAG 2.1 Level AA準拠を目標
- **ストレージ：** 写真は自動圧縮（長辺1920px、500KB以下）。無料枠の範囲内で運用（目安5GB）
- **バックアップ：** 入退場データはCSVエクスポートで手動バックアップ可能

## 8. Technical constraints

- **ユーザー技術レベル：** プロのエンジニア（React得意）
- **フロントエンド：** React（ユーザー指定）
- **インフラ経験：** AWS（業務利用）
- **予算：** 無料枠での運用を最優先。月額コストはKAZASUの3,000円以下
- **デバイス：** 保護者はスマホ（375px〜）、先生はPC＋スマホ、QR読取は施設PC限定
- **フレームワーク・BaaS等の選定は `tech-selection` で決定する**

## 9. Data model sketch

```
User
  - id, email, name, role (parent | teacher | admin), password_hash
  - created_at, updated_at

Child
  - id, name, grade, qr_code (unique), qr_active (bool)
  - created_at, updated_at

ChildParent (join)
  - child_id -> Child
  - parent_id -> User (role=parent)

Attendance
  - id, child_id -> Child, type (enter | exit), method (qr | manual)
  - recorded_at, recorded_by -> User

Announcement
  - id, title, body, posted_by -> User (role=teacher)
  - created_at

AnnouncementRead (join)
  - announcement_id -> Announcement
  - user_id -> User (role=parent)
  - read_at

Photo
  - id, url, thumbnail_url, caption, event_name
  - visibility (private | public), uploaded_by -> User
  - created_at

Document
  - id, title, file_url, category, uploaded_by -> User
  - created_at

BillingRule
  - id, regular_end_time, rate_per_unit, unit_minutes
  - effective_from, created_by -> User

MonthlyBill
  - id, child_id -> Child, year_month
  - total_extended_minutes, total_amount
  - status (draft | confirmed), confirmed_at, confirmed_by -> User

SitePage
  - id, slug, title, content (rich text)
  - updated_at, updated_by -> User

SiteNews
  - id, title, body, published_at
  - created_by -> User

NotificationPreference
  - user_id -> User
  - method (push | email | both | off)
```

## 10. External integrations

- **メール送信：** 招待メール、パスワードリセット、通知フォールバック用（SendGrid、Resend等）
- **プッシュ通知：** Web Push API（PWA対応ブラウザ向け）
- **QRコード生成：** クライアントサイドライブラリ（qrcode.js等）
- **QRコード読取：** ウェブカメラAPI＋クライアントサイドデコーダー（html5-qrcode等）
- **画像ストレージ：** オブジェクトストレージ（S3互換）
- **認証プロバイダ：** tech-selectionで決定（Supabase Auth等が候補）

## 11. Milestones

### M1 — 認証基盤
- アカウント管理（3ロール）、ログイン、パスワードリセット、児童・保護者紐付け
- **検証ポイント：** 保護者がログインして自分の子供を確認できる

### M2 — 入退場管理（KAZASUリプレイス）
- QRコード発行・読取・入退場記録・手動フォールバック・通知・履歴閲覧
- **検証ポイント：** KAZASUと並行運用し、入退場記録の正確性を比較

### M3 — 連絡・共有（LINEリプレイス）
- 連絡配信・通知・写真共有（公開/非公開）・資料掲示板
- **検証ポイント：** 先生が1週間LINEなしで連絡配信を完結できる

### M4 — 請求自動化（紙運用リプレイス）
- 延長料金ルール設定・自動計算・オンライン請求確認・CSVエクスポート
- **検証ポイント：** 1ヶ月分の請求を自動計算し、手計算結果と一致する

### M5 — 外部HP（新規顧客獲得）
- 外部向けHP編集・公開（施設情報・お知らせ・公開写真ギャラリー）
- **検証ポイント：** 入園希望者がHPから説明会情報を見つけて問い合わせる

## 12. Risks & mitigations

| # | リスク | 影響 | 緩和策 |
|---|--------|------|--------|
| 1 | **QR読取の信頼性不足** — ウェブカメラの品質やQRの印刷品質に依存。失敗するとオペレーション停止 | 高 | 手動フォールバック（US-7）を必須実装。移行期間はKAZASUと並行運用 |
| 2 | **iOSプッシュ通知の制限** — PWAのプッシュ通知はiOS Safariで制限あり。保護者の大半がiPhoneの場合、通知が届かない | 中 | メール通知をフォールバックとして実装。通知方法を保護者が選択可能に |
| 3 | **ストレージコスト超過** — 写真の蓄積で無料枠を超える可能性 | 中 | 画像の自動圧縮、容量監視アラート、アーカイブポリシーの策定 |
| 4 | **保護者の移行抵抗** — LINEから新アプリへの移行にITリテラシーの壁 | 中 | シンプルなUX設計。LINEは一時並行運用し段階的移行 |
| 5 | **延長保育ルールの複雑化** — 実運用で例外（早退、イベント延長等）が多発し、単純なルールエンジンでは対応不能 | 低〜中 | v1は基本ルール（通常終了時刻＋固定単価）のみ。例外は手動修正で対応 |

## 13. Open questions

1. **延長保育の計算単位は？** — 30分単位？15分単位？端数の扱い（切り上げ？切り捨て？）
2. **QRコードの形態は？** — カード型（児童が持つ）？壁掲示型（先生がかざす）？
3. **既存KAZASUのデータ移行は必要？** — 過去の入退場記録をインポートするか
4. **写真のストレージ上限は？** — 無料枠の具体的な容量制限と、超過時の対応方針
5. **法人化のスケジュールは？** — HPの優先度・公開タイミングに影響
