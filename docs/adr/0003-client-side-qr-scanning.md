# ADR-0003: QRコード読取をクライアントサイドで完結させる

Date: 2026-04-20
Status: accepted

## Context

入退場管理はgakudoの最も重要な機能（KAZASU代替）。QRコードを読み取ってから画面に結果が表示されるまで3秒以内が要件。施設のPCに接続したウェブカメラでQRを読み取る構成を想定。

QR読取の実装方法として、クライアントサイド完結型（ブラウザのみ）とサーバーサイド画像処理型の2つの選択肢がある。

## Options considered

1. **クライアントサイド完結（html5-qrcode）** — ブラウザのMediaStream APIでカメラ映像を取得し、WebAssembly/JSでQRをデコード。サーバーには読取結果（QRコード文字列）のみ送信。
2. **サーバーサイド画像処理** — カメラ映像のフレームをサーバーに送信し、サーバーでデコード。ネットワーク遅延が加算される。
3. **ネイティブアプリ（カメラAPI）** — 最も高速だが、ネイティブアプリ開発はv1スコープ外。

## Decision

**Option 1: クライアントサイド完結** を採用。

html5-qrcodeライブラリがブラウザ上でQRコードをデコードし、デコード結果のみをServer Actionに送信する。

処理フロー:
1. ブラウザがウェブカメラを起動（html5-qrcode）
2. QRコード検出 → デコード（クライアント側、~100ms）
3. Server Action `recordAttendance(qrCode)` 呼び出し
4. DB: 当日の最新記録から入室/退室を自動判定
5. 画面に結果表示 + 非同期で通知トリガー

入退場の自動判定ロジック:
- 当日の最新記録が「入室」→ 今回は「退室」
- 当日の記録なし or 最新が「退室」→ 今回は「入室」
- 先生は手動で入室/退室を指定可能（オーバーライド）

## Consequences

- **Positive:** ネットワーク遅延がデコードに影響しない。3秒以内の要件を容易に達成。
- **Positive:** オフライン耐性：QR読取自体はネット不要（記録送信のみネット必要）。
- **Positive:** サーバーに画像データを送信しないためプライバシーリスクが低い。
- **Negative:** ブラウザのカメラAPI対応が前提（モダンブラウザは全対応）。
- **Negative:** QR読取不能時の手動フォールバックが別途必要（安全上必須）。
